# Collision-Zone Thinking

Revolutionary insights come from forcing unrelated concepts to collide. Treat X like Y and see what emerges.

**Core principle:** Deliberate metaphor-mixing generates novel solutions.

## When to Use This Skill

- When conventional approaches feel inadequate
- When you need breakthrough innovation, not incremental improvement
- When stuck in domain-specific thinking
- When brainstorming hackathon ideas that need to stand out
- When judges have seen every standard solution and you need something fresh

## Process

1. **Pick two unrelated concepts** from different domains
2. **Force combination**: "What if we treated [A] like [B]?"
3. **Explore emergent properties**: What new capabilities appear?
4. **Test boundaries**: Where does the metaphor break?
5. **Extract insight**: What did we learn?

## Quick Reference

| Stuck On | Try Treating As | Might Discover |
|----------|-----------------|----------------|
| Payment processing | Immune system | Fraud detection as antibodies |
| Supply chain tracking | Nervous system | Real-time reflex responses |
| Patient data flow | Water flow | Streaming health data lakes |
| Credit scoring | DNA sequencing | Behavioral genome risk patterns |
| Last-mile delivery | Ant colony | Swarm optimization routing |
| Code organization | Genetics | Mutation testing, evolution |
| Service architecture | Lego bricks | Composable microservices |
| Error handling | Circuit breakers | Fault isolation patterns |

## Instructions

When a user asks to apply collision-zone thinking:

1. **Identify the problem domain** — What are they stuck on?
2. **Select 3-5 collision sources** from different fields:
   - Biology (immune systems, DNA, evolution, ecosystems)
   - Physics (gravity, thermodynamics, quantum mechanics)
   - Economics (markets, auctions, game theory, incentives)
   - Social Systems (networks, dating, viral spread, reputation)
   - Nature (ant colonies, flocking, weather systems, rivers)
   - Engineering (circuits, bridges, plumbing, HVAC)

3. **Generate collisions** — For each source, create the prompt:
   > "What if we treated [problem] like [source domain concept]?"

4. **Map emergent properties** — For each collision:
   ```markdown
   **Collision:** [Problem] × [Source]
   **Emergent Properties:**
   - [Property 1]: [How it maps]
   - [Property 2]: [How it maps]
   **Where it works:** [Strong mapping areas]
   **Where it breaks:** [Weak mapping areas]
   **Insight gained:** [The novel approach extracted]
   ```

5. **Rank by novelty AND feasibility** — Wild collisions are great, but they must translate to buildable solutions.

## Example Collision

**Problem:** Complex distributed system with cascading failures

**Collision:** "What if we treated services like electrical circuits?"

**Emergent properties:**
- Circuit breakers (disconnect on overload)
- Fuses (one-time failure protection)
- Ground faults (error isolation)
- Load balancing (current distribution)

**Where it works:** Preventing cascade failures
**Where it breaks:** Circuits don't have retry logic
**Insight gained:** Failure isolation patterns from electrical engineering

## Remember

- Wild combinations often yield best insights
- Test metaphor boundaries rigorously
- Document even failed collisions — they teach
- Best source domains: physics, biology, economics, psychology
- Score the UNDERLYING TECH novelty, not just the metaphor cleverness
